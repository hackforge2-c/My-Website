DNS (Domain Name System)

Foundation

DNS is like the phonebook of the internet. Instead of remembering IP addresses like 142.250.183.206, we use names like google.com.

DNS Resolution Flow

Why DNS Matters in Cybersecurity

Attackers often exploit DNS using:

  • DNS Spoofing
  • DNS Cache Poisoning
  • Malicious Domain Redirection

How DNS Works (Simple Flow)

  1. You type a website name
  2. Your system asks DNS resolver
  3. Resolver queries servers
  4. IP is returned
  5. Browser connects

Key DNS Components

🔍
Resolver

Your device's first point of contact. It asks other servers on your behalf to find the IP address.

🌍
Root Server

The top of the DNS tree. There are 13 root server clusters worldwide. They know where TLD servers are.

🗂️
TLD Server

Handles top-level domains like .com, .org, .in. Points to the authoritative server.

🛡️ Security Tip: Always verify HTTPS to avoid DNS-based attacks. Use DNS over HTTPS (DoH) to encrypt your DNS queries so attackers on the network can't intercept them.

DNS Attack Types (Beginner Explanation)

💉
DNS Spoofing / Cache Poisoning

Imagine asking for Google's address and someone gives you a fake address that leads to a malicious website. The resolver stores this fake answer in its cache — so everyone using it gets redirected too.

🎭
Domain Hijacking

Attacker gains control of a domain's DNS settings by compromising the domain registrar account. They redirect all traffic to their own server.

🚇
DNS Tunneling

Attackers hide malicious data inside DNS queries to bypass firewalls. The data is encoded in the domain name itself — e.g. stolensecrets.evil.com.

🌐 Full DNS resolution — iterative query from resolver to root to TLD to authoritative server

HTTP & HTTPS

Critical

HTTP is the protocol used to transfer web data. Every time you open a website, your browser and the server are communicating using HTTP requests and responses.

HTTP Request / Response Cycle

HTTP vs HTTPS

FeatureHTTPHTTPS
Encryption❌ None — plain text✅ SSL/TLS encrypted
Port80443
Data visible?Yes — attacker can readNo — encrypted
Used for?Old/legacy sitesAll modern websites
SecurityVulnerable to MITMProtected

Hackers can intercept HTTP traffic easily → called Man-in-the-Middle attack.

HTTP Methods

📥
GET

Fetch/retrieve data from the server. Like reading a webpage. Data is visible in the URL.

📤
POST

Send data to the server. Used in login forms, file uploads. Data is in the request body — not visible in URL.

✏️
PUT / PATCH

Update existing data on the server. PUT replaces the whole thing; PATCH updates part of it.

🗑️
DELETE

Remove a resource from the server. If not secured properly, attackers can delete data.

🔎
OPTIONS / HEAD

OPTIONS asks what methods are allowed. HEAD gets headers only (no body). Both used in recon by attackers.

Status Codes

CodeMeaningHacker Relevance
200 OKRequest succeededPage exists — potential target
301 RedirectMoved permanentlyCan be abused for open redirects
403 ForbiddenNo permissionResource exists but blocked — try to bypass
404 Not FoundDoesn't existUsed to enumerate hidden pages
500 Server ErrorServer crashedMay reveal stack traces / info leakage
401 UnauthorizedLogin requiredAuth bypass target

Headers (Important for Hackers)

  • Cookies — store session tokens; stealing these = account takeover
  • User-Agent — identifies browser; can be spoofed by attackers
  • Authorization — carries API tokens; leaked tokens = full access
⚠️ Security Insight: Misconfigured headers → vulnerability. Missing security headers like Content-Security-Policy, X-Frame-Options, and Strict-Transport-Security are direct attack vectors.
🔐 HTTPS adds an SSL/TLS encryption layer on top of HTTP — making all data unreadable to interceptors
03

How Websites Work

Core

When you open a website, your browser communicates with a server using HTTP. Understanding this flow is essential because every step is a potential attack surface.

Full Web Request Flow

Frontend

🏗️
HTML

Structure — the skeleton of the webpage. Defines headings, paragraphs, forms, and buttons. Attack: injecting malicious HTML tags.

🎨
CSS

Design — controls colours, layout, fonts. Mostly safe, but can be used in clickjacking and UI redressing attacks.

⚡
JavaScript

Logic — makes websites interactive. The #1 target for XSS attacks. Runs in the user's browser — never trust JS input.

Backend

🖧
Server

Runs code that processes requests. Examples: Node.js, Apache, Nginx. Misconfigured servers expose sensitive files.

🗄️
Database

Stores all data — users, passwords, content. SQL Injection attacks target this layer directly to steal data.

🔗
APIs

The communication bridge between frontend and backend. Broken API authentication is one of the top OWASP risks.

Security Components

  • WAF (Web Application Firewall) — filters and blocks malicious HTTP traffic
  • CDN — distributes content globally; also absorbs DDoS attacks
  • Load Balancer — spreads traffic across multiple servers for availability

Static vs Dynamic

TypeWhat it meansSecurity Risk
StaticFixed content — same HTML served to everyoneLower risk — no database queries
DynamicGenerated per request — pulls from DBHigher risk — SQL injection, XSS, CSRF possible

Common Web Attacks

  • XSS (Cross Site Scripting) — injecting malicious JavaScript into a web page
  • SQL Injection — inserting SQL commands into input fields to manipulate the database
  • CSRF — tricking a logged-in user into performing unwanted actions
⚡ Important: Most cybersecurity attacks happen at the web layer. Understanding how websites work is the foundation of both offensive (pen testing) and defensive (hardening) security work.
🏛️ Web application architecture — user → WAF → load balancer → web server → database
04

Cookies & Sessions

Attack Surface

HTTP is stateless — meaning every request is treated as brand new with no memory of who you are. Cookies and sessions solve this problem by remembering you between requests.

📌 Simple Analogy: When you enter a concert, you get a wristband (cookie). Every time staff checks you, they see the wristband and know you already paid. Without it, you'd have to prove yourself every single time.

What is a Cookie?

A cookie is a small piece of data the server sends to your browser, which the browser stores and sends back with every future request. This lets the server remember you.

🔐
Session Cookie

Deleted when you close the browser. Used for login sessions. If an attacker steals this — they can log in as you without a password. This is called Session Hijacking.

💾
Persistent Cookie

Stays saved even after you close the browser. Used for "remember me" features. Longer lifetime = larger attack window.

🛡️
Secure & HttpOnly Flags

Secure = only sent over HTTPS. HttpOnly = JavaScript cannot read the cookie — prevents XSS from stealing it. Always set both flags on session cookies.

Cookie Security Attacks

AttackWhat happensPrevention
Session HijackingAttacker steals your session cookie via sniffing or XSSUse HTTPS, HttpOnly, short session lifetime
CSRFAttacker tricks your browser into sending a cookie to a malicious siteSameSite cookie flag, CSRF tokens
Cookie Theft via XSSMalicious JS reads document.cookieHttpOnly flag blocks JS cookie access

APIs & REST

Modern Web

An API (Application Programming Interface) is how the frontend and backend communicate. When you press "Login", the browser sends an API request to the server, which checks your credentials and sends back a response.

📌 Simple Analogy: An API is like a waiter in a restaurant. You (frontend) tell the waiter (API) what you want. The waiter goes to the kitchen (backend/database), gets it, and brings it back to you.
REST API Request/Response

Common API Security Issues

🔑
Broken Authentication

APIs that accept expired tokens, weak passwords, or no auth at all. Attacker simply calls the API without logging in.

📂
IDOR (Insecure Direct Object Reference)

Changing /api/user/123 to /api/user/124 and getting someone else's data. No authorisation check on the server side.

💥
Rate Limiting Missing

No limit on how many requests you can make. Attackers brute-force login APIs or scrape entire databases.

🗝️
Exposed API Keys

Developers accidentally commit API keys to GitHub. Attackers scan GitHub for these and get full access to services.

Web Attacks Deep Dive

High Priority

The three most common and dangerous web attacks that every cybersecurity learner must know in detail.

XSS — Cross Site Scripting

XSS happens when an attacker injects malicious JavaScript code into a webpage that other users then load in their browsers. The browser executes the script thinking it is trusted.

📌 Simple Example: A website shows comments posted by users. An attacker posts a comment containing <script>alert('hacked')</script>. Every user who loads that page executes the attacker's script.

XSS Types:

💾
Stored XSS (Persistent)

Malicious script is stored in the database and served to every user who loads that page. Most dangerous type — one injection affects all users.

🔗
Reflected XSS

Script is in the URL. Attacker sends victim a malicious link. When victim clicks it, the script executes once in their browser.

🧠
DOM-based XSS

Happens entirely in the browser — the server never sees it. JavaScript reads from a dangerous source like location.hash and writes it directly to the DOM.

Prevention: Encode all output. Use Content-Security-Policy header. Never use innerHTML with user data.

SQL Injection

SQL Injection happens when an attacker inserts SQL commands into an input field (like a login form) that the server executes directly against the database.

Classic example: In a login form, username field receives: ' OR '1'='1 — this makes the SQL query always true and logs the attacker in without a valid password. 
-- Normal query:
SELECT * FROM users WHERE username='admin' AND password='1234';

-- After injection:
SELECT * FROM users WHERE username='' OR '1'='1' --' AND password='';

Prevention: Use prepared statements / parameterised queries. Never build SQL strings by concatenating user input. Use an ORM.

CSRF — Cross Site Request Forgery

CSRF tricks a logged-in user's browser into sending a request to a website without the user knowing. The browser automatically attaches the victim's cookies — so the server thinks it is a legitimate request.

📌 Simple Example: You are logged into your bank. You visit a malicious site that has a hidden form that sends a transfer request to your bank. Your browser submits the form with your session cookie. The bank processes the transfer.

Prevention: Use CSRF tokens on all state-changing requests. Use the SameSite cookie attribute. Check the Origin header on the server.

💉 XSS attack flow — attacker injects script → victim loads page → browser executes malicious code

Security Headers

Defence

Security headers are HTTP response headers that tell the browser how to behave when handling your website's content. They are one of the easiest and most powerful defences to implement.

HeaderWhat it doesProtects against
Content-Security-PolicyDefines which sources of scripts, images and styles are trustedXSS, Data injection
Strict-Transport-SecurityForces browser to always use HTTPSMITM, SSL stripping
X-Frame-OptionsPrevents your site from being embedded in an iframeClickjacking
X-Content-Type-OptionsPrevents browser from guessing file typesMIME sniffing attacks
Referrer-PolicyControls how much referrer info is sharedInformation leakage
Permissions-PolicyControls access to browser features (camera, mic, location)Feature abuse
💡 Quick Check: You can test any website's security headers for free at securityheaders.com. Missing headers will be flagged — this is a standard part of any web pen test.

Hacker Tools for Web

Pen Testing

These are the most commonly used tools in web application security testing. Every ethical hacker needs to know these.

🔥
Burp Suite

The #1 web hacking tool. Acts as a proxy between browser and server — intercepts, modifies, and replays HTTP requests. Industry standard for web pen testing.

🕷️
OWASP ZAP

Free alternative to Burp Suite. Automatically scans websites for common vulnerabilities. Great for beginners and used in bug bounty hunting.

🗺️
Nikto

Web server scanner. Detects dangerous files, outdated software, misconfigurations, and default credentials on web servers automatically.

📁
DirBuster / Gobuster

Directory and file brute-forcers. Discover hidden paths on a website like /admin, /backup.zip, /.env by trying thousands of common names.

💉
SQLMap

Automatically detects and exploits SQL injection vulnerabilities. Can dump entire databases, read files, and sometimes get a shell on the server.

🌊
Sublist3r / Amass

Subdomain enumeration tools. Find hidden subdomains like dev.target.com, api.target.com — often less secured than the main site.
